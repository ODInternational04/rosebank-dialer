# Changelog: April 19 - April 21, 2026

## Overview
This document summarizes all changes made to the Rosebank Dialer system between April 19 and April 21, 2026. The focus has been on **security hardening**, **production readiness**, and **user experience improvements** for login authentication.

---

## 🔐 Security Enhancements

### 1. Production Security Baseline
- **New Document**: `SECURITY_BASELINE_ALL_PLATFORMS.md`
  - Comprehensive security guidelines covering:
    - Application security controls
    - API security measures
    - Supabase security best practices
    - Vercel deployment security
    - GitHub repository security
    - Operational security procedures
  - Release gate checklist for production deployments
  - Emergency hardening procedures

### 2. Debug/Test Endpoint Protection
**All debug and test endpoints are now blocked in production:**

- `/api/debug/*` routes:
  - `create-user`
  - `customer-feedback`
  - `data`
  - `fix-admin`
  - `stats`
  - `test-reports`
  - `users`

- `/api/test/*` routes:
  - `call-data`
  - `customer-feedback`
  - `database-setup`
  - `db`
  - `email-config`
  - `email-notification`
  - `feedback-insert`
  - `hash`
  - `login`

**Implementation**: All endpoints return `404 Not Found` in production environment.

### 3. Authentication Security Improvements

#### Enhanced Login Lockout System
- **Reduced maximum attempts**: From 5 to **3 attempts**
- **Lockout duration**: Changed to **10 minutes** (was 15 minutes)
- **Real-time attempt tracking**: Users now see remaining attempts
- **Lockout timer**: Visual countdown showing when account will unlock
- **Persistent state**: Lockout state persists across page refreshes

#### Admin Notifications
- Admins are automatically notified via email when accounts get locked
- Notification includes:
  - Target email address
  - Source IP address
  - Number of failed attempts
  - Lock expiration time

#### Rate Limiting Updates
- **App-level rate limit increased**: 30 requests per 5 minutes (was 5)
- Credential-specific lockout now handled at application level
- Better error responses with `RATE_LIMIT_EXCEEDED` code

### 4. CORS Security
- **Production CORS hardening**:
  - Auth endpoints now use strict origin validation
  - Removed wildcard (`*`) CORS for authentication
  - Configurable via `NEXT_PUBLIC_APP_URL` environment variable
  - Default origin: `https://rosebank-dialer.vercel.app`

### 5. Password Policy Enforcement
- **Strengthened password requirements**:
  - Minimum 12 characters (was 6)
  - Must include uppercase letter
  - Must include lowercase letter
  - Must include number
  - Must include special character
- Applied to user registration and user creation endpoints

### 6. Email Normalization
- All email addresses are now **normalized to lowercase**
- Prevents duplicate accounts with different casing
- Applied to:
  - User registration
  - Login attempts
  - Email lookups

### 7. SMTP Security
- **Production TLS enforcement**: `rejectUnauthorized: true` in production
- Self-signed certificates only allowed in development

## 🎨 User Experience Improvements

### Login Page Enhancements

#### Visual Feedback
1. **Attempt Counter**: Shows remaining login attempts after failed login
   - Displayed in amber warning banner
   - Updates in real-time

2. **Lockout Timer**: Visual countdown when account is locked
   - Format: `MM:SS` (e.g., `9:45`)
   - Red danger banner with countdown
   - Updates every second

3. **Persistent State**: 
   - Email address remembered across page refresh
   - Attempt count persists
   - Lockout timer persists

4. **Disabled Input During Lockout**:
   - Email and password fields disabled when locked
   - Button shows lockout timer
   - Prevents unnecessary form submissions

#### Error Messages
- More descriptive error responses:
  - `ACCOUNT_LOCKED`: Too many failed attempts
  - `RATE_LIMIT_EXCEEDED`: Rate limit exceeded
  - `INVALID_CREDENTIALS`: Invalid email or password
- Each error includes appropriate retry information

## 🐛 Bug Fixes

### 1. False "Failed to create user" Responses
- **Issue**: Supabase insert succeeded but select/return failed, causing false error
- **Fix**: Added fallback verification query
  - If insert errors, check if user was actually created
  - Return success (201) if user exists with matching email
  - Prevents duplicate error states

### 2. Missing Supabase Client Initialization
- **Issue**: Several API routes referenced `supabase` without initializing it
- **Fixed routes**:
  - `/api/call-logs/[id]/route.ts` (DELETE)
  - `/api/clients/search/route.ts` (GET)
  - `/api/notifications/[id]/route.ts` (DELETE)
  - `/api/notifications/route.ts` (PUT, DELETE)
  - `/api/statistics/route.ts` (POST)

### 3. Login Context Issues
- **AuthContext improvements**:
  - Better error handling and response structure
  - Returns detailed `LoginResult` object
  - Includes error codes, retry information, and attempts remaining
  - Properly handles all authentication states

## 📝 API Response Changes

### Login Endpoint (`/api/auth/login`)

#### New Response Fields
```json
{
  "error": "string",
  "code": "INVALID_CREDENTIALS | ACCOUNT_LOCKED | RATE_LIMIT_EXCEEDED",
  "attemptsRemaining": 0-3,
  "retryAfter": number (seconds)
}
```

#### Status Codes
- `200`: Successful login
- `401`: Invalid credentials (with attempts remaining)
- `429`: Account locked or rate limited
- `404`: Debug/test endpoint access in production

## ⚠️ Breaking Changes

### 1. Password Requirements
**Old**: Minimum 6 characters  
**New**: Minimum 12 characters + complexity requirements

**Action Required**: Existing users with weak passwords should be prompted to update on next login.

### 2. Login Attempts
**Old**: 5 attempts before lockout  
**New**: 3 attempts before lockout

**Impact**: Users with incorrect credentials will be locked out faster.

### 3. Debug/Test Endpoints
**Old**: Accessible in all environments  
**New**: Return 404 in production

**Impact**: Production monitoring/debugging must use proper logging and analytics tools.
